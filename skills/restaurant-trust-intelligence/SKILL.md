# Restaurant Trust Intelligence

이 스킬은 단순 맛집 검색용이 아닙니다. 맛집 추천을 공식 가이드, 맛집러 직접 언급, 광고/협찬 리스크, 운영 정보, 사용자 취향, 실제 방문 후보로 분리해 관리합니다.

## 기본 원칙

- 현재 맛집 정보는 변하기 쉬우므로, 최신 추천을 줄 때는 반드시 최신 출처를 확인합니다.
- 미슐랭, 빕 구르망, 블루리본은 기본 공식 가이드 레이어로 봅니다. 서로 같은 의미로 취급하지 말고 각각 분리해 표시합니다.
- “맛집러가 추천했다”는 말은 해당 맛집러의 공식 콘텐츠에서 직접 확인된 경우에만 씁니다.
- 스타일이 비슷하다, 좋아할 것 같다, 댓글에서 봤다, 블로그가 요약했다는 이유만으로 직접 추천이라고 쓰지 않습니다.
- 광고, 협찬, 초대, 제공, 캠페인 가능성이 있으면 추천 이유보다 먼저 표시합니다.
- 맛집러 목록은 고정이 아닙니다. 추가, 승격, 강등, 보류, 삭제가 가능합니다.
- 내 취향과 비슷한 맛집러, 또는 나보다 더 고급 취향을 가진 맛집러를 우선 추적합니다.

## 강한 추천 시그널

다음 표현이나 맥락은 높은 가치의 추천 신호입니다.

- 해당 분야 최고
- 국내 1위
- 장르 1위
- 인생 맛집
- 자주 가는 단골
- 무조건 경험해볼 만한 곳
- 극강의 가성비, 단 실제 퀄리티가 동반될 때
- 장인의 퀄리티
- 비교 영상이나 랭킹에서 최종 상위권
- 고가 식당을 제치고 저가 식당이 월간/연간 상위권에 오른 경우

단, 이런 표현이 광고성 콘텐츠에만 반복되면 신뢰도를 낮춥니다.

## 맛집러 선정 기준

좋은 맛집러:

- 내 취향과 비슷하다.
- 나보다 훨씬 고급 취향을 가지고 있어 학습 가치가 있다.
- 추천 이유와 메뉴 설명이 구체적이다.
- 단골, 재방문, 비교, 랭킹, 실패 경험을 함께 말한다.
- 광고/협찬을 명확히 표시한다.

제거 또는 보류할 맛집러:

- 단순 홍보, 바이럴, 마케팅 콘텐츠 비중이 높다.
- 편의점, 프랜차이즈, 제품 리뷰가 많아 식당 추천 목적과 맞지 않는다.
- 술집/안주 중심인데 사용자가 술을 선호하지 않는다.
- 맛보다 분위기, 핫플, 조회수 중심이다.
- 추천 식당을 따라갔을 때 사용자 피드백이 반복적으로 나쁘다.

## YouTube Shorts 처리 규칙

Shorts는 1급 근거로 취급합니다. 다만 제목만 믿지 않습니다.

Shorts에서 반드시 확인할 것:

- 제목
- 설명란
- 해시태그
- 업로드일
- 자동 자막 또는 화면 텍스트
- 식당명 별칭
- 지점명
- 주소/지역명

예: 영상 제목만으로는 식당명이 드러나지 않지만, 설명란이나 해시태그에 실제 식당명, 가격, 주소가 들어 있을 수 있습니다.

따라서 Shorts와 릴스 중심 채널은 다음처럼 처리합니다.

- 제목은 식당명 추출용으로 신뢰하지 않습니다.
- 제목은 추천 강도와 감정 신호로 봅니다.
- 식당명, 가격, 주소, 메뉴는 설명란과 해시태그에서 우선 추출합니다.
- 식당 없는 해명성 영상, 수익 공개, 잡담 영상은 제외합니다.

## 교차검증 규칙

한 맛집러에게서 식당 하나가 발견되면, 1군 맛집러 전체와 공식 가이드 레이어에서 역검색합니다.

역검색은 표준 식당명뿐 아니라 별칭, 지점명, 지역명, 장르명까지 사용합니다.

예:

- 표준 식당명
- 지점명
- 지역명 + 식당명
- 지역명 + 장르명
- 대표 메뉴명

서초점과 부산점처럼 지점 경험이 다를 수 있는 경우 자동 병합하지 않습니다.

상태값:

- `confirmed`: 공식 콘텐츠에서 직접 확인
- `user-reported pending`: 사용자가 제보했지만 아직 링크 확인 전
- `searched/no-hit`: 어느 범위까지 검색했는지 기록한 미확인
- `branch-separated`: 같은 브랜드지만 지점별로 분리 관리

## 표준 추천 양식

추천은 기본적으로 다음 형식을 따릅니다.

```markdown
최신 확인 기준: YYYY-MM-DD
범위/조건:
주의: 운영상태, 휴무, 예약 가능 여부는 방문 전 재확인 필요.

## 가격대/조건 Top N

| 순위 | 식당 | 3줄 추천 | 메뉴별 추천 사유 | 가격대 | 위치 | 출처/주의 |
|---:|---|---|---|---|---|---|
| 1 | 식당명 | 1) 강한 추천 신호. 2) 내 취향과 맞는 이유. 3) 언제 가면 좋은지. | `메뉴 A`: 왜 대표 메뉴인지. `메뉴 B`: 무엇을 확인하는 메뉴인지. |  |  |  |
```

규칙:

- 모든 식당에는 최소 3줄 추천 이유가 있어야 합니다.
- “맛있음”, “유명함”, “가성비 좋음”만 쓰지 않습니다.
- 주요 메뉴마다 추천 사유를 씁니다.
- 코스요리, 카포, 스시야, 오마카세는 주요 코스와 접시별 특징을 씁니다.
- 코스 세부 정보가 없으면 `코스 세부 미확인`으로 표시하고 순위를 낮추거나 보류합니다.
- 맛집러 근거와 AI의 부가 분석을 분리합니다.

## 일일 업데이트 절차

1. 공식 가이드 변경 사항을 확인합니다.
2. 1군 맛집러의 최신 영상, Shorts, 인스타그램, 지도/리스트를 확인합니다.
3. 새 식당 후보를 식당 원장부에 추가합니다.
4. 광고/협찬 리스크를 표시합니다.
5. 식당별 메뉴, 가격, 위치, 운영 정보를 보강합니다.
6. 사용자 방문 후기가 있으면 `visit-feedback.md`에 기록하고, `taste-profile.md`와 `creator-registry.md`에 반영합니다.
7. 사용자 취향에 맞춰 방문 후보 큐를 재정렬합니다.
8. 새로 발견한 식당은 1군 맛집러 전체에 대해 역검색합니다.
9. 변경 내역을 `daily-update-log.md`에 기록합니다.

## 답변 원칙

- 출처, 신뢰도, 광고 리스크, 취향 적합성, 운영 정보를 분리해 말합니다.
- 직접 확인되지 않은 추천은 직접 추천처럼 쓰지 않습니다.
- 최신 확인이 필요한 경우 반드시 최신 확인 필요라고 표시합니다.
- 사용자가 “왜 이걸 가야 하냐”고 물었을 때, 메뉴별로 설득 가능한 수준까지 씁니다.
- 사용자의 피드백은 다음 추천의 취향 모델에 반영합니다.
- 방문 후기는 맛집러 신뢰도와 사용자 취향을 동시에 업데이트하는 근거로 사용하되, 둘을 한 점수로 합치지 않습니다.
